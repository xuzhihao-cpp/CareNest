## 验收依据

- 项目工作目录：`smart-nursing-platform/`
- 阶段任务卡：`docs/team/phase-26-31-optimized-ai-task-cards.md`

- 边界：偏好护理只表达家属意愿，不等于最终派单，不在前端创建护理任务或修改订单状态。

## 前端实现

### 家属预约

- 阶段29推荐卡增加“设为偏好”操作。
- 仅 `available=true` 的当前推荐候选可选择，不可预约候选按钮禁用。
- 服务对象、服务、地址或时间变化时，旧偏好随推荐上下文一起清空。
- 相同预约条件下重新获取推荐时，会再次核对当前偏好；护理已不在最新结果或已不可用时立即清空，不能携带旧偏好下单。
- 读取 `GET /auth/permissions`，只有家属具备 `NURSE_PREFERENCE_SELECT` 时才开放偏好选择；权限读取失败只关闭偏好功能，不阻断普通下单。
- 读取真实 `GET /family/bindings`，目标长辈必须存在 `ACTIVE` 绑定且生效范围包含 `ORDER_CREATE`；绑定失效、待确认或缺少代下单授权时关闭预约与偏好入口。
- 授权提示由当前选中长辈实时计算；初始化并行读取时不会因长辈尚未选中而误报“没有代下单授权”。
- 新订单通过阶段10创建请求携带 `preferredNurseId`。
- 提交使用独立单次执行锁，请求完成前按钮显示“正在提交”，连续点击不会重复创建订单。
- 新订单创建成功后重新读取订单列表和订单详情，不以创建响应冒充最终订单状态。
- 下单预览显示偏好护理姓名，不显示护理内部编号。

### 已有订单

- 家属订单列表读取并显示偏好护理姓名。
- 订单列表同时显示后端返回的中文推荐依据，不只显示护理姓名。
- 订单详情通过 `GET /family/orders/{orderId}/recommendation-view` 读取当前偏好，并与订单推荐结果关联为护理姓名。
- 仅 `WAIT_DISPATCH` 可修改偏好；其他状态自动只读。
- 修改使用 `PUT /family/orders/{orderId}/preferred-nurse`，成功后重新读取偏好视图、推荐结果和订单详情。
- `409` 会提示订单状态已变化并重新读取，不保留可编辑假状态。
- 订单详情读取使用请求序号和订单主键双重校验，快速查看多笔订单时旧响应不会覆盖当前详情。
- 家属订单列表使用独立请求序号；初始化、跨页更新事件和偏好更新并发时仅最新列表响应可写入页面。

### 管理端

- 管理端统一读取 `GET /auth/permissions`；只有具备 `NURSE_RECOMMEND_VIEW` 才挂载订单推荐区、偏好推荐依据和派单面板。
- 缺少权限或权限读取失败时不会调用护理推荐接口，并显示业务化的只读提示。
- 管理端订单详情和派单工作台增加“家属偏好护理”只读摘要。
- 摘要突出护理姓名与推荐依据，同时说明管理员仍需按实际情况选择合格护理员。
- 管理端不调用仅限家属角色的偏好视图接口，不越权模拟结果。
- 管理端读模型只返回护理姓名和推荐理由、没有内部编号时，仍可完整展示家属偏好。
- 订单接口未提供偏好字段时，页面明确显示暂未读取到偏好，不把推荐第一名冒充家属选择。
- 订单读模型已提供 `preferredNurseName` 和 `preferredNurseReason` 时优先直接展示；即使护理后来离开当前推荐列表或推荐接口暂时失败，列表和详情仍稳定显示订单保存时的偏好信息。

## 接口与数据

- `PUT /api/v1/family/orders/{orderId}/preferred-nurse`
  - 请求：`{preferredNurseId}`
  - 响应：`{orderId,preferredNurseId,recommendReason}`
- `GET /api/v1/family/orders/{orderId}/recommendation-view`
  - 响应同上；无偏好返回 `404`。
- `GET /api/v1/orders/{orderId}/recommendations`
  - 用于把偏好护理内部关联值解析成用户可读姓名和推荐依据。

页面不展示 `preferredNurseId`、API、DTO、追踪编号、数据库字段名或英文状态枚举。

阶段30承载链路中的阶段10、11、12 API 已移除运行时 fixture、`isMockEnabled()` 分支及 `mock:` 回退参数，只调用真实后端接口。

## 当前真实联调阻塞

### 1. 新订单没有保存偏好护理

当前 `FamilyOrderRequest` 接收 `preferredNurseId`，但后端创建订单 SQL 没有写入 `nursing_order.preferred_nurse_id`，也没有在创建瞬间重新校验候选资格、培训和可用性。前端会正确提交字段，但刷新后无法读取到该偏好。

### 2. 管理端订单读模型缺少偏好字段

当前管理端订单详情 DTO 不包含 `preferredNurseId`、护理姓名和推荐原因；阶段30偏好视图又只允许家属角色访问。因此管理端无法真实展示家属已经选择的偏好。

需要在现有管理端订单详情读模型中增加可展示字段，推荐至少返回 `preferredNurseName` 和 `preferredNurseReason`，避免管理端二次接触内部编号。

### 3. 缺少独立的全量可派护理列表契约

前端已经移除阶段12的静态护理名单，改为读取当前订单真实推荐结果，并只把 `available=true` 的候选提供给管理员明确选择；不会自动选择家属偏好或推荐第一名。若业务要求管理员查看推荐结果之外的全部合格护理，当前冻结接口仍缺少独立的全量可派护理列表契约。

### 4. 继承阶段29的推荐契约阻塞

后端推荐原因仍可能包含 `BASIC_CARE` 等技术编码，护理技能字典接口也尚未形成完整冻结契约。阶段30不会在前端改写理由或硬编码技能名称。
