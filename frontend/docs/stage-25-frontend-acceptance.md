# 阶段 25 前端验收记录：服务前健康摘要

## 实现边界

1. 护理端从待接单、已接单或前往中的当前已派任务进入服务前健康摘要，不允许手工输入订单编号。服务中暂不开放入口，后端契约明确允许前不得自行放开。
2. 页面只调用 `GET /api/v1/nurse/orders/{orderId}/pre-service-health-summary`，不调用健康档案编辑、病历审核或报告写入接口。
3. 展示顺序固定为重点风险、过敏信息、当前用药、慢病与照护要点、审核通过病历、近期服务摘要。
4. 页面不展示订单 ID、长辈 ID、文件 ID、档案版本、存储路径、内部审核意见、接口地址或 traceId。
5. 病历资料仅在响应提供真实授权预览地址时显示预览操作。同源受保护地址以 Bearer Token 读取为临时 Blob；跨域仅接受签名 URL，且不得携带登录令牌。
6. 页面无输入控件，不允许在护理端直接修改健康档案。
7. 服务名称缺失时显示“上门护理服务”，不得回退展示 `serviceId` 或其他内部编码。

## 隐私与状态

1. 每次打开、切换订单或刷新摘要时，先清空当前摘要，再请求新订单数据。
2. 请求使用递增序号和当前 `orderId` 双重校验；旧订单迟到响应不得回写。
3. 返回任务列表和组件卸载时立即清空摘要并使在途请求失效。
4. 同源病历预览使用 `AbortController`；刷新、切换任务、返回列表或卸载组件时取消旧文件请求并关闭尚未完成的空白预览窗口。
5. `401/403/404/409/502` 显示中文业务提示，不回退本地数据或假成功。
6. 成功响应必须逐项校验档案、病历和报告数据；集合元素不完整时按 `502` 处理，不交给模板渲染。
7. 空风险、过敏、用药、慢病、病历或报告均显示对应真实空状态，不生成示例内容。

## 自动化与真实联调门禁

- `pnpm test:stage25`：检查固定接口、Bearer 登录态、响应完整性、真实 403、业务格式化、只读页面、展示顺序和跨订单隔离。
- `pnpm typecheck`：Vue 与 TypeScript 类型检查。
- `pnpm build:h5`：护理端 H5 正式构建。
- 真实联调还必须验证：被派护理人员可读，未派护理人员及用户角色返回 403，阶段 24 归档后刷新可见，未审核病历不在响应中，后端记录摘要访问日志。

阶段 25 前端不包含 mock。后端尚未部署该接口时，页面必须显示真实读取失败，不能以空摘要代替成功。
