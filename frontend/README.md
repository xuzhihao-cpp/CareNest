# frontend

前端目录，后续承载长辈端、家属端、护理端和管理端页面。
移动端采用uni-app包括长辈端、家属端、护理端。
web端包含管理端。

## 阶段 19 健康档案写入约定

- 完整健康档案编辑统一使用 `PUT /api/v1/elders/{elderId}/health-archive`，一次提交慢病、用药、过敏、风险标签和照护计划，并携带当前档案版本。
- `POST /api/v1/elders/{elderId}/medications` 仅保留给独立的“快速新增用药”交互。当前完整编辑页面不调用该接口，避免一次保存被拆成多次写入或产生版本冲突。
- 两种写入方式都必须在成功后重新读取健康档案；不得只更新前端本地状态。

## 阶段 20 病历资料上传约定

- 家属上传采用两步流程：先通过 `POST /api/v1/files` 上传文件，再通过 `POST /api/v1/elders/{elderId}/medical-files` 登记资料类别、标题和发生日期。
- 若文件上传成功但资料登记失败，页面只在当前会话内暂存 `fileId`，允许用户直接重新登记；刷新页面后不保留该临时状态，也不会制造成功记录。
- 病历资料列表始终通过 `GET /api/v1/elders/{elderId}/medical-files` 重新读取。页面展示待审核、已通过、未通过和需补充四种服务端状态及审核意见。待审核以阶段文档的 `PENDING_REVIEW` 为正式契约，读取时兼容仓库字典中的 `PENDING`；需补充同样兼容 `NEED_MORE` 与 `NEEDS_SUPPLEMENT`。
- 家属查看需要有效绑定和健康查看权限，上传还需要健康编辑权限；长辈端仅提供只读查询。
- 当前支持 PDF、JPG、JPEG 和 PNG。单文件默认上限为 10 MB，可通过 `VITE_MEDICAL_FILE_MAX_MB` 与后端部署限制保持一致。
- 仅在后端返回真实预览或下载地址时显示对应操作。删除、替换和分享未纳入当前接口契约，因此阶段 20 页面不提供这些入口。
- 阶段 20 不使用本地 mock、`localStorage`、Base64 文件持久化或前端假成功。后端接口未部署时必须显示真实失败状态。
- 前端自动化检查和真实后端联调门禁见 `docs/stage-20-frontend-acceptance.md`。

## 阶段 21 管理端病历审核约定

- 管理员或客服进入页面后必须先通过真实 `GET /api/v1/auth/permissions` 读取病历审核权限；权限读取失败或缺少审核权限时不加载病历。客服账号不会显示其他管理功能。
- 列表、详情和审核分别使用真实 `GET /api/v1/admin/medical-files`、`GET /api/v1/admin/medical-files/{fileId}` 和 `POST /api/v1/admin/medical-files/{fileId}/review`，不初始化本地病历或回退假成功。
- 默认读取待审核资料，支持服务端分页，以及状态、资料类型、长辈姓名和日期筛选。页面不显示文件内部编号、接口路径、DTO 或 traceId。
- 审核支持通过、驳回和要求补充。驳回或要求补充必须填写审核意见；当前前端仅允许处理待审核资料，重复处理由后端返回 `409` 后刷新最新状态。审核成功后即使原记录退出“待审核”筛选，也会按原记录重新读取最新详情。
- “进入档案审核流程”只随通过结论提交后续档案建议，不表示健康档案已经修改。正式档案写入仍由阶段 24 完成。
- 前端兼容 `PENDING_REVIEW`/`PENDING` 和 `NEEDS_SUPPLEMENT`/`NEED_MORE` 两组读取状态，并统一使用中文业务状态显示。
- 阶段 21 自动化检查和真实后端联调门禁见 `docs/stage-21-frontend-acceptance.md`。

## 阶段 22 长辈健康反馈约定

- 长辈端通过真实 `POST /api/v1/elder/health-feedback` 提交反馈，身份只取当前登录态，请求体不接受或携带 `elderId`。
- 反馈覆盖疼痛、头晕、睡眠、饮食和精神状态，程度分为轻微、明显和严重，输入分为快捷、文字和语音。
- 语音先通过 `POST /api/v1/files` 上传。上传成功但反馈登记失败时保留当前会话内的 `fileId`，允许直接重新登记，避免重复上传。
- 家属端仅对 `ACTIVE + HEALTH_VIEW` 的绑定调用 `GET /api/v1/family/elders/{elderId}/health-feedback`，并按类型、程度、日期和分页读取真实时间线。
- 严重反馈只提供克制的风险提醒和现有家属联系方式入口，不生成诊断、用药调整或“已通知他人”等未经后端确认的结果。
- 家属快速切换长辈时使用请求版本与资源标识共同阻止旧响应回写。同源受保护语音携带登录令牌读取为临时 Blob；可信跨域签名语音不附带令牌，其他来源拒绝播放。
- `view=emergency` 对应真实协助视图，提供家属联系方式、可配置的平台热线和 120 急救电话操作；不自行新增未冻结的求助接口，也不显示虚假的通知成功。
- 阶段 22 不使用本地 mock、假时间线或前端假成功。自动化检查和真实后端联调门禁见 `docs/stage-22-frontend-acceptance.md`。

## 阶段 23 健康档案变更建议约定

- 护理端通过当前真实任务、服务记录和服务报告选择建议来源，不允许手工填写订单编号、来源编号或内部字段名。
- 建议值沿用阶段 19 的慢性病、当前用药、过敏记录、健康风险和照护计划结构，并在提交前执行对应字段校验。
- 提交只调用 `POST /api/v1/orders/{orderId}/health-update-suggestions`。成功表示建议进入管理端审核，不表示健康档案已经修改。
- 健康建议使用 `PENDING / APPROVED / REJECTED / ARCHIVED`，审核任务使用 `PENDING / IN_REVIEW / ARCHIVED / REJECTED`，两套状态不得相互转换。
- 管理员或客服先通过 `GET /api/v1/auth/permissions` 核对健康档案审核权限，再调用 `GET /api/v1/admin/health-review-tasks` 读取真实建议队列。
- 管理端以长辈和服务为业务标题，对照展示档案当前内容与建议内容，不显示技术编号、接口路径或原始对象文本。
- 建议原因最长 255 字；任一真实来源接口异常时显示对应失败，不得以“暂无记录”掩盖权限或服务错误。
- 重复建议、来源不匹配和越权均保留后端真实 `409`、`422` 或 `403` 结果；阶段 23 不创建本地审核记录，也不调用阶段 19 档案写入接口。
- 自动化检查和真实后端联调门禁见 `docs/stage-23-frontend-acceptance.md`。

## 阶段 24 健康信息审核归档约定

- 阶段 24 是护理建议和审核病历写入正式健康档案的唯一审核入口，前端不得用阶段 19 的档案编辑接口代替归档。
- 管理员或客服具备健康审核权限后，通过真实任务详情核对来源证据、当前值、建议值和服务端规范化值，再逐字段选择采纳归档、不采纳或需要补充。
- 归档请求只调用 `POST /api/v1/admin/health-review-tasks/{taskId}/archive`；不采纳和需要补充必须填写说明。服务端返回 `409` 时重新读取原任务，不显示本地成功。
- 归档完成后重新读取任务详情、任务队列和真实变更日志，并展示最新任务状态与档案版本。
- 长辈和家属健康档案页通过 `GET /api/v1/elders/{elderId}/health-archive/change-logs` 展示可读历史，不显示审核员、日志或来源内部编号。
- 阶段 24 不使用 mock、本地审核状态或前端假变更记录。自动化检查和真实联调门禁见 `docs/stage-24-frontend-acceptance.md`。

## 阶段 25 服务前健康摘要约定

- 护理端从待接单、已接单或前往中的当前已派任务进入摘要，通过真实 `GET /api/v1/nurse/orders/{orderId}/pre-service-health-summary` 读取数据，不允许手工填写订单编号。服务中状态在后端契约明确允许前不显示摘要入口。
- 页面只读并优先展示重点风险、过敏信息和当前用药，随后展示慢病与照护要点、审核通过病历和近期服务摘要。
- 每次打开、刷新、切换或离开任务时先清空旧摘要，并用请求序号与订单资源共同阻止迟到响应串到另一位长辈页面；同时取消未完成的同源病历预览请求。
- 病历资料只显示后端返回的授权预览入口；同源受保护地址携带登录令牌读取为临时 Blob，跨域仅打开签名 URL且不附带登录令牌。
- 页面不显示订单或长辈内部编号、服务编码、档案版本、文件存储路径、审核意见、接口地址或 traceId，不提供任何健康档案编辑入口。服务名称缺失时使用中文通用名称。
- 成功响应会逐项校验长辈档案、用药、慢病、过敏、病历和报告数据，缺少模板依赖字段时按响应不完整处理。
- 阶段 25 不使用 mock、假摘要或本地成功状态。自动化检查和真实联调门禁见 `docs/stage-25-frontend-acceptance.md`。
